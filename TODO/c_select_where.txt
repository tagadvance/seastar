PLAN: SELECT WHERE / column projection / LIMIT filtering (extend SelectHandler)

CURRENT STATE: SelectHandler ignores the select clause and WHERE entirely and returns every row.
Goal: honor projection, WHERE equality/IN, and LIMIT. Match Cassandra's query-restriction rules.

IMPLEMENTATION
1. Projection: if selectClause is non-empty, build ColumnDefinitions with only the selected columns
   (in selected order) and project each row down to those columns. "*" keeps all columns.
   newAsyncResultSet(columnDefinitions, executionInfo, rows) already takes explicit ColumnDefinitions.
2. WHERE: implement equality (=) and IN first. For each relation, resolve the bound/literal value,
   find the column index, and keep rows whose value equals (any of) the target(s).
   Apply LIMIT after filtering.
3. Bindings: marker values come from bindings[bindIndex]; literals parsed from the Term.Raw.

FIDELITY (verify against ContainerCqlSessionTest -- Cassandra is strict here)
  - Filtering on a non-primary-key column without ALLOW FILTERING -> InvalidQueryException
    ("Cannot execute this query as it might involve data filtering..."). Mirror this: allow WHERE on
    partition/clustering key columns; require parameters.allowFiltering for other columns.
  - Unknown column in WHERE or SELECT -> InvalidQueryException.
  - Invalid limit (<=0) -> InvalidQueryException.
  Keep unsupported operators (ranges on partition keys, token(), etc.) throwing
  UnsupportedOperationException with a clear message, and log DISTINCT-not-supported (already done).
